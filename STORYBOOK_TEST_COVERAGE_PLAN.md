# План покрытия компонентов тестами через Storybook

## Контекст
- [x] Проект: mars.client (JavaScript SDK проект [MARS.client.esproj](MARS.client.esproj)).
- [x] Storybook уже подключен и используется вместе с Vitest (`@storybook/addon-vitest`) и Playwright browser mode.
- [x] В проекте уже есть набор story-файлов (OBS, shared, dashboard, admin), что позволяет строить тесты поверх существующих сценариев.

## Цель
Сделать Storybook основным источником компонентных тестов: каждый критичный UI-сценарий описан через story и автоматически проверяется в CI.

## Целевые метрики
- [ ] Критичные компоненты (P0): 100% покрытие story-тестами happy-path + минимум 1 негативный кейс.
- [ ] Важные компоненты (P1): не менее 80% покрытие основных состояний (loading/empty/error/data).
- [ ] Общий порог компонентного покрытия (Vitest coverage):
  - [ ] lines >= 60%
  - [ ] branches >= 50%
  - [ ] functions >= 60%
- [ ] Accessibility smoke: для всех P0 и P1 stories должен проходить `@storybook/addon-a11y`.

## Область покрытия

### P0 (в первую очередь)
- [ ] Scoreboard и AdminPanel блоки.
- [ ] SoundRequest Player (desktop/mobile) и связанные экраны (CurrentTrack, TrackList, VideoScreen).
- [ ] ChatVertical/ChatHorizontal и Message-компоненты.
- [ ] Dashboard метрики, если используются в прод-потоке управления стримом.

### P1 (второй приоритет)
- [ ] RandomMem, WaifuAlerts, PyroAlerts, AutoMessageBillboard, AFKScreen.
- [ ] Shared-компоненты с высокой переиспользуемостью: CRUDPage, PageName, KeyWordText, InjectStyles.

### P2 (по мере необходимости)
- [ ] Визуальные/декоративные компоненты без бизнес-риска (анимации, экспериментальные эффекты).

## Стратегия тестирования внутри Storybook

### 1) Базовый стандарт для каждой story
- [ ] У каждой критичной компоненты должны быть минимум stories:
  - [ ] `Default` (базовый рендер)
  - [ ] `Loading`
  - [ ] `Empty`
  - [ ] `Error`
  - [ ] `WithData` (реальный рабочий сценарий)
- [ ] Для интерактивных компонентов добавить `play`-функции с пользовательскими действиями (click/input/keyboard).

### 2) Проверки в `play`
- [ ] Проверка видимости ключевых элементов интерфейса.
- [ ] Проверка перехода состояния после действия пользователя.
- [ ] Проверка вызовов локальных обработчиков (если прокинуты через args).
- [ ] Проверка boundary-случаев (пустые данные, длинный текст, отсутствующие поля).

### 3) A11y-проверки
- [ ] На уровне stories использовать тесты доступности из `addon-a11y`.
- [ ] Для P0/P1 сценариев зафиксировать минимальный набор правил:
  - [ ] контраст,
  - [ ] aria-label для контролов,
  - [ ] корректные роли для интерактивных элементов.

### 4) Моки данных
- [ ] Вынести фикстуры рядом с компонентами (`__fixtures__`) или в `src/shared/test-data`.
- [ ] Для каждого домена (scoreboard, sound-request, chat) иметь минимальный стабильный mock payload.
- [ ] Избегать случайных/временных данных в stories, чтобы тесты не флапали.

## План внедрения по этапам

## Этап 1. Аудит и нормализация stories (1-2 дня)
- [ ] Собрать матрицу: компонент -> какие состояния уже есть -> чего не хватает.
- [ ] Довести P0-компоненты до полного набора состояний (`Default/Loading/Empty/Error/WithData`).
- [ ] Убрать дублирующиеся истории и нестабильные моки.

## Этап 2. Интерактивные тесты для P0 (2-4 дня)
- [ ] Добавить `play`-сценарии для всех P0 stories.
- [ ] Добавить проверки a11y для P0.
- [ ] Запустить локально `yarn test` и исправить флак.

## Этап 3. Расширение до P1 (2-3 дня)
- [ ] Повторить подход для P1-компонент.
- [ ] Добавить недостающие empty/error кейсы.
- [ ] Привести названия stories к единому шаблону (`ComponentName/State`).

## Этап 4. CI quality gate (1 день)
- [ ] В CI запускать минимум:
  - [ ] `yarn lint:noFix`
  - [ ] `yarn test --run`
  - [ ] `yarn test:coverage`
  - [ ] `yarn build-storybook`
- [ ] Падать сборкой при нарушении порогов покрытия.
- [ ] Публиковать отчеты coverage как артефакт.

## Этап 5. Поддержка и регрессии (постоянно)
- [ ] Для каждого багфикса UI добавлять story-кейс, воспроизводящий баг до фикса.
- [ ] Для каждой новой компоненты требовать story + минимальный play-test до merge.
- [ ] Раз в спринт пересматривать покрытие P0/P1 и список критичных компонентов.

## Чек-лист для Pull Request
- [ ] Добавлены/обновлены stories для измененных компонентов.
- [ ] Для интерактивной логики есть `play`-проверки.
- [ ] Есть минимум 1 негативный сценарий.
- [ ] Локально проходят `yarn test` и `yarn build-storybook`.
- [ ] Нет случайных/временных данных, вызывающих нестабильность.

## Definition of Done
- [ ] Все P0 компоненты покрыты story-тестами по целевому стандарту.
- [ ] Не менее 80% P1 покрыто основными состояниями.
- [ ] CI стабильно (минимум 3 последовательных успешных прогона без flaky падений).
- [ ] Пороги покрытия выполняются и зафиксированы в pipeline.

## Быстрые команды
```bash
yarn storybook
yarn test --run
yarn test:coverage
yarn build-storybook
```
